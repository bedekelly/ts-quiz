TypeScript Questions

    Basic Types §

  1. What type is this? 
    5
    It's a 
    number
    .
  2. What type is this? 
    "hello"
    It's a 
    string
    .
  3. Is this a number? 
    "5"
    No, it's a 
    string
    .
  4. Is this a number? 
    15
    Yes.
  5. Is this a number? 
    -1
    Yes.
  6. Is this a number? 
    0.5
    Yes.
  7. Is this a string? 
    ""
    Yes.

    8. Arrays (1) §

  8. What type is this? Don't worry about syntax for now. 
    [1, 2, 3]
    It's an array of numbers.
  9. What type is this? Don't worry about syntax for now. 
    ["hello", "world"]
    It's an array of strings.
  10. What type is this? Don't worry about syntax for now. 
    ["hello", 123, "world", 456]
    It's an array of strings and numbers.

    11. Objects §

  11. What type is this? 
    { a: 123 }
    It's an object.
  12. What type is this? 
    { a: [1, 2, 3]}
    It's an object.
  13. What type is this? 
    { a: ["hello", "world"], b: [4, 5, 6]}
    It's an object.
  14. What type is this? 
    { a: { b: ["hello"]}, b: [1, 2, { c: "world" }]}
    It's an object. (We see a problem emerging. Calling all of these "an object" is a bit too vague.)

    15. Objects and Keys §

  15. What keys does this object have? 
    { a: 1, b: 2 }
    The keys are 
    a
    and 
    b
    .
  16. What keys does this object have? 
    {}
    It doesn't have any keys.
  17. What keys does this object have? 
    { a: 1, b: { c: 3 }}
    The keys are 
    a
    and 
    b
    . Look carefully: 
    c
    is not a key of this object.

    18. Objects and Values §

  18. What type are the values of this object? 
    { a: 1, b: 2 }
    The values are of type 
    number
    .
  19. What type are the values of this object? 
    { a: 'hello', b: 'world' }
    The values are of type 
    string
    .
  20. What type are the values of this object? 
    { a: 1, b: 'hello', c: 2, d: 'world' }
    The values are of type 
    string
    and 
    number
    .

    21. Object Types §

  21. Is it true that this object 
    a
    is of type 
    { hello: string }
    ? 
    const a = {
    hello: 'world'
}
    Yes: the TypeScript compiler by default remembers exactly which keys are in an object, but the values are degraded to more generic types.
  22. Is it true that this object 
    a
    is of type 
    { hello: string, myNumber: number }
    ? 
    const a = {
    hello: 'world',
    myNumber: 123
}
    Yes; again, the compiler remembers the keys 
    hello
    and 
    myNumber
    perfectly, but remembers the more generic types 
    string
    and 
    number
    for the values.

    23. Annotations §

  23. Is it true that this 
    a
    is a number? 
    const a = 5
    Yes, 
    a
    here is a number.
  24. Is it true that 
    a
    is a number? 
    const a: number = 5
    Yes. We often don't need to, but we can add a type annotation here.
  25. Is it true that this 
    a
    is of type 
    number
    ? 
    const a = 5
    Yes. This is just a slightly more precise way of saying the same thing.
  26. Is it true that this 
    a
    is of type 
    number
    ? 
    const a: number = 5
    Yes. This is just a slightly more precise way of saying the same thing.
  27. Is it true that 
    a
    is of type 
    5
    ? 
    const a: 5 = 5
    Yes. We've instructed the compiler to be even more specific with what type it remembers for this value. Instead of remembering that this value is a generic number, it makes a note that this value is the literal number 5.

    28. Annotations (2) §

  28. Is it true that 
    a
    is of type 
    10
    ? 
    const a = 10
    No. Without that annotation, the TypeScript compiler doesn't assume we want the literal number. For most cases, it's enough to remember that it's a 
    number
    .
  29. Will this compile without errors? 
    const expects_string: string = 5
    No, since 
    5
    does not match the type of 
    string
    . The compiler tells us 
    Type 'number' is not assignable to type 'string'
    . 
  30. my_number
    is of type 
    number
    . Will this compile without errors? 
    const expects_five: 5 = my_number
    No. 
    expects_five
    has a more specific type, and the compiler can't prove that 
    my_number
    will meet the requirements of that type. It tells us 
    Type 'number' is not assignable to type '5'
    . If the compiler allowed this, we'd be able to set 
    expects_five
    to 
    4
    in this situation, or 
    10,000
    , or 
    -8.5
    .
  31. Is it true that 
    a
    is of type 
    'hello'
    ? 
    const a: 'hello' = 'hello'
    Yes. We can make more specific types with strings too!

    32. Subtypes §

  32. Is the value 
    1
    assignable to type 
    number
    ?
    Yes, 1 is a value which matches 
    number
    .
  33. Is the value 
    "hello"
    assignable to type 
    number
    ?
    No, 
    "hello"
    is not a valid value for 
    number
    .
  34. Is the value 
    "hello"
    assignable to type 
    number | string
    ?
    Yes, 
    "hello"
    is a valid value for 
    number | string
    , because it's a valid value for 
    string
    .
  35. Is the value 
    123
    assignable to type 
    number | string
    ?
    Yes, 
    123
    is a valid value for 
    number | string
    , because it matches 
    number
    .
  36. Is it true that this 
    a
    is of type 
    'hello' | 'world'
    ? 
    const a: 'hello' | 'world' = 'hello'
    Yes.
  37. Is it true that this 
    a
    is of type 
    'hello' | 'world'
    ? 
    const a: 'hello' = 'hello'
    No. The compiler only remembers what types it's told about, so 
    a
    is of type 
    'hello'
    .

    38. Subtypes (2) §

  38. Will this compile without errors? 
    let a: string
const b: 'hello' = 'hello'
a = b;
    Yes, the compiler allows more specific types to be assigned to variables with less specific types.
  39. Will this compile without errors? 
    let a: 'hello' | 'world';
const b: 'hello' = 'hello';
a = b;
    Yes, the compiler allows more specific types to be assigned to variables with less specific types.
  40. Will this compile without errors? 
    function returnsHelloOrWorld(): 'hello' | 'world' {
    return 'hello';
}
let a: 'hello' = returnsHelloOrWorld()
    No, the compiler will report 
    'hello' | 'world'
    is not assignable to type 
    'hello'
    . For all it knows, the returned value is 
    'world'
    .
  41. Will this compile without errors? 
    function returnsHelloOrWorld() {
    return 'hello';
}
let a: 'hello' = returnsHelloOrWorld()
    No. Without a return type annotation, the compiler infers that the function returns a 
    string
    , and doesn't allow it to be assigned to a variable with a more specific type 
    'hello'
    .
  42. Will this compile without errors? 
    function returnsHelloOrWorld(): 'hello' {
    return 'hello';
}
let a: 'hello' = returnsHelloOrWorld()
    Yes, the annotation tells the compiler that the function returns 
    'hello'
    , so it can be assigned to the variable 
    a
    .
  43. Will this compile without errors? 
    function returnsHelloOrWorld(): 'hello' {
    return 'world';
}
let a: 'hello' = returnsHelloOrWorld()
    No, the compiler checks that the return value matches the annotation on a function.

    44. Arrays (2) §

  44. What type is this? 
    [1, 2, 3]
    An array of numbers. In TypeScript we can write this as 
    Array<number>
    .
  45. What type is this? 
    ["hello", "world"]
    An array of strings. In TypeScript we can write this as 
    Array<string>
    .
  46. What type is this? 
    [1, "hello", 2, "world"]
    An array of strings and numbers. In TypeScript we can write this as 
    Array<string | number>
    .
  47. What type is 
    list
    here? 
    const a: 'hello' = 'hello';
const b: 'world' = 'world;
const list = [a, b];
     
    list
    is of type 
    Array<'hello' | 'world'>
    .
  48. Is 
    [1, 2, 3]
    of type 
    number[]
    ?
    Yes, 
    number[]
    is another way of writing 
    Array<number>
    .
  49. Is 
    ["hello", "world"]
    of type 
    string[]
    Yes, 
    string[]
    is another way of writing 
    Array<string>
    .
  50. Is 
    list
    here of type 
    ('hello' | 'world')[]
    ? 
    const a: 'hello' = 'hello';
const b: 'world' = 'world;
const list = [a, b];
    Yes, 
    list
    is of type 
    Array<'hello' | 'world'>
    , which can also be written as 
    ('hello' | 'world')[]
    . Round brackets here are just used to group types together.
  51. Is 
    Array<number>
    the same type as 
    number[]
    ?
    Yes, 
    Array<something>
    and 
    something[]
    are two ways of writing the same thing. They refer to exactly the same type.
  52. Is there a value which can be an 
    Array<number>
    , but not a 
    number[]
    ?
    No, because the two types are the same, they have exactly the same possible values.

    53. Promises §

  53. Is 
    Promise.resolve(15)
    of type 
    Promise<number>
    ?
    Yes, 
    Promise<number>
    is a type which means a promise which resolves to a number.
  54. What type is 
    a
    here? 
    async function f() {
	return 123;
}
const a = f();
     
    a
    is a 
    Promise<number>
    . Async functions return promises which resolve to the values they return. This has the same effect as writing 
    Promise.resolve(123)
    .
  55. What type is 
    a
    here? 
    async function f() {
	return Promise.resolve(123);
}
const a = f();
     
    a
    is a 
    Promise<number>
    . Nested promises get "flattened".
  56. What type is 
    a
    ? 
    const a = Promise.resolve(Promise.resolve(Promise.resolve(123)))
     
    a
    is a 
    Promise<number>
    . Nested promises get "flattened".

    57. Types with parameters §

  57. What type is 
    a
    ? 
    const a = Promise.resolve([1, 2, 3]);
     
    a
    is a 
    Promise<Array<number>>
    . It's a promise of an array of numbers.
  58. What type is 
    a
    here? 
    async function f() {
	return [1, 2, 3]
}
const a = f()
     
    a
    is a 
    Promise<Array<number>>
    . It's a promise of an array of numbers.
  59. Is 
    Promise<Array<number>>
    the same type as 
    Promise<number[]>
    ?
    Yes, because 
    Array<number>
    is the same type as 
    number[]
    .
  60. What type is 
    a
    ? 
    const h: 'hello' = 'hello';
const a = Promise.resolve(h);
     
    a
    is a 
    Promise<'hello'>
    . It's a promise which resolves to a value of type 
    'hello'
    .
  61. Is 
    Promise.resolve("hello")
    a 
    Promise<'hello'>
    ?
    No, by default the compiler will only infer strings as 
    string
    , not anything more specific. 
    Promise.resolve("hello")
    is a 
    Promise<string>
    .

    62. Nested Arrays §

  62. What type is 
    a
    ? 
    const a = [
	[1, 2, 3],
	[4, 5, 6]
];
     
    a
    is of type 
    Array<Array<number>>
    . It's an array of arrays of numbers.
  63. Is 
    a
    an 
    Array<Array<string>>
    ? 
    const a = [
	["hello", "world"]
]
    Yes, 
    a
    is an 
    Array<Array<string>>
    .
  64. Is 
    a
    a 
    string[][]
    ? 
    const a = [
	["hello", "world"]
]
    Yes, 
    a
    is an 
    Array<Array<string>>
    , which can also be written as 
    string[][]
    .
  65. What type is 
    a
    here? 
    async function f() {
	return [
		["hello", "world"]
	]
}
const a = f();
    Yes, 
    a
    is a 
    Promise<string[][]>
    , or alternatively a 
    Promise<Array<Array<string>>>
    . It's a Promise which will resolve to a value of type 
    string[][]
    .

    66. Aliases §

  66. Is 
    NumberAlias
    a type alias for 
    number
    ? 
    type NumberAlias = number
    Yes, 
    NumberAlias
    is now a type alias for 
    number
    .
  67. Is 
    NewAlias
    a type alias for 
    number
    ? 
    type NewAlias = NumberAlias
    Yes, setting one alias for another will set the new alias to the original type.
  68. Will this compile without errors? 
    type NumberAlias = number
const a: NumberAlias = 123;
    Yes, 
    NumberAlias
    is an alias for 
    number
    , so this is the same as writing 
    const a: number = 123
    .
  69. Will this compile without errors? 
    type NumberAlias = number
type NewAlias = NumberAlias
const a: NewAlias = 123;
    Yes, 
    NewAlias
    is another alias for 
    number
    , so this is the same as writing 
    const a: number = 123
    .
  70. Will this compile without errors? 
    type NumberAlias = number
type NewAlias = NumberAlias
const a: NewAlias = 'hello';
    No, 
    NewAlias
    is another alias for 
    number
    , so this is the same as writing 
    const a: number = 'hello'
    .
  71. Is there a value which matches 
    NumberAlias
    which doesn't match 
    number
    ?
    No, 
    NumberAlias
    is exactly the same type as 
    number
    , so all the same values will match both of them.

    72. Aliases for types with parameters §

  72. What is type 
    Mystery
    here? 
    type A = number
type B = Array<A>
type Mystery = Promise<B>
     
    Mystery
    is an alias for 
    Promise<Array<number>>
    , or equivalently 
    Promise<number[]>
    . Aliases don't create any new types, they're just new names for existing ones that can save some typing.
  73. What is type 
    Puzzle
    here? 
    type A = 'hello' | 'world'
type B = Array<Promise<A>>
type Puzzle = B | A
     
    Puzzle
    is of type 
    Array<Promise<'hello' | 'world'>> | 'hello' | 'world'
    . Aliases are replaced like-for-like with the types they refer to.
  74. Does 
    'hello'
    match the type 
    Puzzle
    above?
    Yes, 
    'hello'
    is one of the possible values for type 
    Array<Promise<'hello' | 'world'>> | 'hello' | 'world'
    .
  75. Does 
    [ Promise.resolve('world') ]
    match the type 
    Puzzle
    above?
    Yes, that's one of the possible values for type 
    Array<Promise<'hello' | 'world'>> | 'hello' | 'world'
    .
  76. Does this compile without errors? 
    type A = Array<number>
type B = Promise<string>
const a: A | B = [ Promise.resolve('123') ];
    No, 
    [ Promise.resolve('123') ]
    is of type 
    Array<Promise<string>>
    , which isn't assignable to either 
    Array<number>
    or 
    Promise<string>
    , so therefore isn't assignable to 
    Array<number> | Promise<string>
    , which is written here with aliases as 
    A | B
    .

    77. Aliases with parameters §

  77. What is type 
    Mystery
    here? 
    type MakeArray<T> = Array<T>
type Mystery = MakeArray<number>
     
    Mystery
    is the type 
    Array<number>
    .
  78. What is type 
    Puzzle
    here? 
    type MakePromise<TypeParameter> = Promise<TypeParameter>
type Puzzle = MakePromise<number>
     
    Puzzle
    is a 
    Promise<number>
    . The type 
    number
    is passed to the parameter 
    TypeParameter
    .
  79. In 
    type MyType<MyParam> = ...
    , is 
    MyParam
    a type parameter?
    Yes.
  80. In 
    type MyType<OneParam, AnotherParam> = ...
    , how many type parameters are there?
    Two: 
    OneParam
    and 
    AnotherParam
    .
  81. In 
    type A<T> = ...
    , is 
    A
    a parameterised type?
    Yes, because it has a parameter 
    T
    , 
    A
    is a parameterised type.

    82. Aliases for types with parameters (2) §

  82. What is type 
    A
    here? 
    type SomeType<ParamOne, ParamTwo> = Array<ParamOne | ParamTwo>;
type A = SomeType<number, string>
    Type 
    A
    is 
    Array<number | string>
    . 
    number
    is passed in as 
    ParamOne
    , and 
    string
    is passed as 
    ParamTwo
    .
  83. Do type parameters need to be a single character?
    No, as in the examples above, type names can be any number of characters.
  84. Does this compile without errors? 
    type SomeType<ParamOne, ParamTwo> = Array<ParamOne | ParamTwo>;
type A = SomeType<number, string>
const a: A = [1, 2, "hello"]
    Yes, 
    a
    is an 
    Array<string | number>
    , so it matches the type referenced by the alias 
    A
    .
  85. Does this compile without errors? 
    type SomeType<ParamOne, ParamTwo> = Array<ParamOne | ParamTwo>;
type A = SomeType<number, string>
const a: A = [1, 2, 3]
    Yes, 
    a
    is a 
    number[]
    , so it matches the type referenced by the alias 
    A
    .
